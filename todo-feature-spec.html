To-Dos — The Third Event Type

How to-dos differ from events and bills, how they're created, how they behave over time, and how the AI briefing picks them up.

How the AI Briefing Handles To-Dos

The briefing mentions to-dos naturally alongside events and bills — not as a separate list, but woven into the prose. It escalates language for overdue items.
Briefing with to-do context
12:05🔋
Calendar
✦ This week · 2–8 Mar
Packed Monday — don't forget the car
Dentist, drama rehearsal, and the electricity bill all land today. That Amazon return is two days overdue — the post office on Condamine is open till 5. Car service needs booking before Wednesday. Thursday brekkie with Krish, then a quieter finish to the week.
AI prompt rule
How to mention to-dos
Mention to-dos in the briefing when they're due soon (within 2 days) or overdue. Use natural language: "that Amazon return is overdue" not "To-do item: Amazon return (overdue)". For overdue items, be specific about how many days. For flexible to-dos with no due date, only mention them if the agenda is light and there's an opportunity: "quiet Wednesday — good chance to look into those internet providers."
Headline picks up urgency
To-dos can drive the headline
If the most notable thing this week is an overdue to-do or an important upcoming one, the headline can reference it: "Packed Monday — don't forget the car". The AI should weigh overdue to-dos as high-priority items alongside bills and big events when choosing what to highlight.

Data Model

To-do fields and how they map to the urgency states.
TodoItem
id: UUID
title: String             // "Book car service"
emoji: String             // "🚗"
startDate: Date?          // nil = starts immediately
dueDate: Date?            // nil = no deadline (flexible)
isCompleted: Bool         // toggles on checkbox tap
completedDate: Date?      // when it was marked done
createdDate: Date         // for "added N days ago"
showInAgenda: Bool        // toggle in creation form
remindWhenDue: Bool       // local notification on due date
Computed: urgencyState
notStarted → startDate is in the future
active → startDate passed, due date > 2 days away (or no due date)
dueSoon → due date within 2 days
overdue → due date has passed, not completed
done → isCompleted = true
Sorting in agenda
To-dos sort after timed events and bills within a day. Among to-dos: overdue first (oldest overdue at top), then due-soon, then active, then flexible. This ensures the most urgent items bubble up without displacing time-specific events.

Key Design Decisions

01
No partial completion in v1
It's tempting to add "in progress" states for multi-day tasks like "organise the garage". But that adds complexity with low payoff. Binary done/not-done is enough. If a task has sub-steps, the user can create multiple to-dos.
02
Carry forward, don't duplicate
An undone to-do doesn't create copies on each new day — it's one item that persists in the agenda until completed. The "days overdue" counter is computed from the due date, not from copies.
03
Teal accent, red for overdue
To-dos use teal (the exercise accent) which works here because it's distinct from event purple and bill amber. When overdue, the accent flips to red — the only time red appears in the system. This makes overdue items impossible to miss.
04
Same + button, third type
The existing + button opens with a type selector at the top: Event / Bill / To-Do. The form below adapts: events show time fields, bills show amount, to-dos show start date + due date + toggles. Same entry point, different forms.
05
Flexible to-dos don't nag
A to-do with no due date shouldn't feel urgent. It sits quietly in the agenda with a grey "Flexible" badge. The AI only mentions it when the day is light: "Quiet afternoon — good chance to research those internet providers."
06
Done items fade, then clear
When you tap the checkbox, the to-do gets a strikethrough + reduced opacity (same treatment as past events). It stays visible for the rest of the day as a "done" item, then clears from the agenda the next morning. Never deleted from history.
